# Kufar rental monitor — как это устроено сейчас

Сервис мониторит поиск аренды квартир на `re.kufar.by` (Гомель) и шлёт
уведомления о новых объявлениях в Telegram, с ценой, числом комнат,
расстоянием до центра и цветной оценкой "горячести".

## Архитектура

```
GitHub Actions (раз в 5 мин, ненадёжно)   Cloudflare Cron Trigger (раз в 5 мин)
        │  обычный GET-запрос (curl)              │  scheduled()
        ▼                                          ▼
Cloudflare Worker "kufarresearch"  ──►  re.kufar.by (SEARCH_URL)
        │        │                          │
        │        │                     HTML + __NEXT_DATA__ (JSON)
        │        ▼
        │   Workers KV (KUFAR_KV, ключ "seen_ids")
        │        │  список уже виденных id объявлений
        ▼
  Telegram Bot API (sendMessage) ──► чат с ботом KufarChecker
```

Планировщик — **и GitHub Actions, и Cloudflare Cron Trigger одновременно**
(почему снова оба — см. раздел "История: почему не Cloudflare Cron, а потом
снова Cloudflare Cron" ниже). Вся остальная логика (парсинг, сравнение,
отправка) — в одном Cloudflare Worker.

## Файлы в репозитории

| Файл | Роль |
|---|---|
| `worker.js` | Весь код воркера (единственный файл с логикой) |
| `wrangler.toml` | Конфиг Cloudflare Worker (biндинги, переменные, KV) |
| `.github/workflows/poll.yml` | Планировщик — дёргает URL воркера каждые 5 минут |
| `README.md` | Инструкция по деплою |
| `STATUS.md` | Этот файл — что как работает сейчас |

## Откуда берутся данные

`re.kufar.by` рендерит страницу поиска на сервере (SSR) и встраивает
все объявления в JSON внутри `<script id="__NEXT_DATA__">`. Воркер:

1. Скачивает `SEARCH_URL` (`fetchSearchHtml`) с заголовками обычного
   Chrome-браузера (`User-Agent`, `Accept-Language: ru-RU`).
   Дополнительно передаёт `cf: { cacheTtl: 0 }` — Kufar сам стоит за
   Cloudflare, и без этого можно случайно получить закэшированную
   (устаревшую) версию страницы вместо свежей.
2. Ищет `__NEXT_DATA__`, парсит его как JSON, рекурсивно ищет внутри
   массив объектов с полем `ad_id` (`findAdsArray` — не завязан на
   точный путь типа `props.pageProps...`, чтобы пережить редизайн
   Kufar без переписывания кода).
3. Для каждого объявления достаёт (`buildAdDetails`):
   - `rooms` — из `ad_parameters` (параметр `rooms`);
   - `coordinates` — `[долгота, широта]`, тоже из `ad_parameters`;
   - `address` — из `account_parameters` (параметр `address`);
   - `priceUsd` / `priceByn` — из `ad.calculator` (цены хранятся в
     центах, отсюда `/ 100`).
4. **Фолбэк**: если `__NEXT_DATA__` не нашёлся или не распарсился —
   `extractAds` откатывается на regex по ссылкам вида
   `/vi/{регион}/snyat/kvartiru/{id}` (и вариант с доп. сегментом
   `v-novostrojke/{id}`). В этом случае есть только id и ссылка, без
   цены/комнат/адреса.

## Логика diff и уведомлений (`runMonitor`)

1. Если Kufar ответил не `200` — считается "заблокировано", в KV
   ничего не пишется, отчёт показывает статус и кусок ответа.
2. Иначе — собираются все id со страницы (обычно ~30, сортировка
   Kufar — `lst.d`, то есть по дате размещения, свежие сверху).
3. Читается `seen_ids` из KV:
   - если ключа вообще нет (`null`) — это **первый запуск**: все
     найденные id сохраняются как база, уведомления **не** шлются
     (иначе на старте улетел бы залп из ~30 сообщений);
   - иначе новые = найденные минус уже виденные.
4. По каждому новому id — сообщение в Telegram (`formatAdMessage`).
   Ошибки отправки не прерывают цикл, копятся в отчёте.
5. **Запись в KV происходит только если что-то изменилось**
   (первый запуск или есть новые id) — экономит дневную квоту записи
   Workers KV (1000/сутки на free-тарифе) при частом опросе.
   `seen_ids` обрезается до последних `MAX_SEEN_IDS` (800) записей.

## Формат уведомления

```
🟢 300р(100$), 2 комн., 2.7 км до центра
3-я Авиационная ул, 9, Гомель, Гомельская область
https://re.kufar.by/vi/1075685401
```

- Цена — сначала BYN, потом USD в скобках (если есть обе валюты).
- Кружок — оценка "горячести" (`classifyHotness`):
  - **цена за комнату** (цена / кол-во комнат, студии считаются за 1
    комнату) — `≤120$` максимум баллов, `≤200$` средне, выше — 0;
  - **расстояние до центра** — считается честно по формуле
    гаверсинуса от координат объявления до `CITY_CENTER` (примерная
    точка центра Гомеля, `52.4245, 31.0017` — **не проверена
    геодезически, это грубая оценка**, можно скорректировать);
    `≤2 км` максимум, `≤5 км` средне, дальше — 0.
  - 🟢 — хорошая сумма баллов (≥75%), 🟡 — средне (≥25%), 🔴 — плохо,
    ⚪ — не хватило данных для оценки (нет цены/комнат и координат).
- Все пороги и координата центра — константы в начале `worker.js`
  (`PRICE_PER_ROOM_GREAT_USD`, `PRICE_PER_ROOM_OK_USD`,
  `CENTER_CLOSE_KM`, `CENTER_OK_KM`, `CITY_CENTER`) — правь напрямую
  в коде, если пороги не устраивают.

## Служебные URL (только для ручной диагностики)

Без авторизации — доступны всем, у кого есть ссылка. Для личного
проекта это осознанно оставлено просто, но ссылку лучше не публиковать.

- `https://kufarresearch.calm4.workers.dev/` — обычный прогон:
  отчёт (найдено/новых id), реально шлёт в Telegram и пишет в KV.
- `.../?debug=1` — показывает, нашёлся ли `__NEXT_DATA__` и кусок
  JSON вокруг первого найденного id. Не трогает KV и Telegram.
- `.../?resetLast=N` — удаляет последние N id из `seen_ids` в KV
  (по умолчанию 5), чтобы на следующем прогоне они снова считались
  "новыми" и ушли в Telegram — способ проверить весь пайплайн без
  ожидания реального нового объявления. Сам ничего не отправляет,
  только чистит базу — отправка произойдёт на следующем срабатывании
  (GitHub Actions, максимум ~5 минут) либо сразу, если открыть
  обычную ссылку вручную.

## Планировщик — GitHub Actions

`.github/workflows/poll.yml`:

```yaml
on:
  schedule:
    - cron: "*/5 * * * *"
  workflow_dispatch: {}
jobs:
  poll:
    runs-on: ubuntu-latest
    steps:
      - run: curl -sS -w "\nHTTP_STATUS:%{http_code}\n" "https://kufarresearch.calm4.workers.dev/"
```

Раз в 5 минут раннер GitHub делает обычный `curl` на публичный URL
воркера — с точки зрения Kufar это неотличимо от обычного захода в
браузере. Можно запустить и вручную — кнопка **Run workflow** во
вкладке **Actions** репозитория (или `workflow_dispatch`).

Репозиторий **публичный** — это важно: на публичных репозиториях
GitHub Actions на стандартных раннерах бесплатны без ограничения по
минутам. На приватном репозитории при интервале в 5 минут вышло бы
~8600 минут/месяц — в разы больше бесплатного лимита в 2000 мин/мес.

## Cloudflare: конфигурация (`wrangler.toml`)

- `name = "kufarresearch"` — должно совпадать с именем Worker-проекта
  в дашборде Cloudflare (иначе Git-деплой ругается на mismatch).
- `[observability] enabled = true` — логи в Cloudflare **Observability
  → Logs** (иначе настройка из дашборда слетает при следующем
  Git-деплое, если её нет в конфиге).
- `[[kv_namespaces]]` — биндинг `KUFAR_KV` на реальный namespace id.
- `[vars]` — `SEARCH_URL` (город/фильтры поиска), `MAX_SEEN_IDS`.
- Секреты `TELEGRAM_BOT_TOKEN` и `TELEGRAM_CHAT_ID` — **не в файле**,
  заданы через `wrangler secret put` / дашборд Cloudflare
  (Settings → Variables and Secrets, тип **Secret**).
- **Cron Trigger в Cloudflare снова включён** — `[triggers] crons =
  ["*/5 * * * *"]` в `wrangler.toml`. Он вызывает `scheduled()` в
  `worker.js`, которая делает то же самое, что и обычный `fetch()`.

Деплой автоматический: любой пуш в `main` пересобирает и выкатывает
воркер через Cloudflare Workers Builds (Git-интеграция), вручную
`wrangler deploy` запускать не нужно.

## История: почему не Cloudflare Cron Trigger, а потом снова Cloudflare Cron

При отладке обнаружилось: запросы к Kufar, инициированные
Cloudflare Cron Trigger, стабильно получали `403` с кастомной
страницей Kufar "Доступ ограничен" — притом что **тот же код**,
вызванный обычным HTTP-запросом (ручной заход в браузере, curl,
внешний планировщик) — всегда получал `200`. Это подтверждено
логами (`scheduled: status=403 ...` на каждом тике в течение 12+
часов подряд, при том что ручные заходы в это же время работали).

Похоже, Cloudflare Cron Triggers исполняются из отдельного,
ограниченного пула дата-центров/IP, который Kufar (временно или
постоянно — доподлинно не известно) заблокировал. Попытка обойти
это самозапросом воркера к самому себе (`fetch(SELF_URL)` внутри
`scheduled()`) была заблокирована уже самим Cloudflare (ошибка 1042 —
защита от бесконечных циклов), даже не дойдя до Kufar.

Решением стал вынос планирования за пределы инфраструктуры Cloudflare:
GitHub Actions делает обычный внешний HTTP-запрос, который выглядит для
Kufar как обычный визит, а не как трафик от известного "кронового" пула.
Cron Trigger был удалён из `wrangler.toml` и из дашборда Cloudflare.

**Проблема:** после переезда на GitHub Actions уведомления в фоне всё
равно не приходили. Диагностика через GitHub API (`actions_list` /
`list_workflow_runs`) показала: у `poll.yml` за всё время существует
**ровно один** прогон, и это `workflow_dispatch` (ручной), ни одного
`schedule`-события GitHub ни разу не прислал — при том что прошло
значительно больше 5-минутных интервалов, чем должно было. `schedule` в
GitHub Actions — best-effort: он может задерживаться или подолгу не
срабатывать вовсе, особенно у свежесозданного workflow в нечасто
активном публичном репозитории — гарантий на срабатывание нет. При
этом Cron Trigger в Cloudflare тоже был удалён (подтверждено в
дашборде: секция Trigger events пустая) — то есть в этом состоянии
**фон не работал вообще ничем**, только ручные заходы на URL.

Решение — снова включить Cloudflare Cron Trigger (`[triggers] crons =
["*/5 * * * *"]` в `wrangler.toml`) **в дополнение** к GitHub Actions,
а не вместо него: два независимых планировщика вместо одного
непроверенного. Прошлый лог со `scheduled: status=200 ...` (пойманный
уже после того, как триггер должен был быть удалён) показывает, что
блокировка IP-пула кронов Kufar сейчас, похоже, не действует — риск
получить `403` снова есть, но раз GitHub Actions `schedule` ненадёжен
сам по себе, лучше иметь оба источника, чем ни одного. Если блокировка
вернётся — в Observability будут видны ошибки `scheduled blocked`,
и в это время должен подстраховать GitHub Actions (когда он всё-таки
сработает).

Есть и обратная сторона: у обоих планировщиков нет блокировки/CAS на
общий ключ KV `seen_ids` (`getSeenIds`/`saveSeenIds` — обычный
read-modify-write). Если Cron Trigger и GitHub Actions сработают
почти одновременно, оба могут прочитать один и тот же `seen_ids`, оба
посчитать одно и то же объявление "новым" и отправить по нему
уведомление дважды — считаем это приемлемым компромиссом (дубль лучше
тишины), но это известное ограничение, не баг, который тут исправлен.

## Известные ограничения / что можно доточить

- `CITY_CENTER` — координаты центра Гомеля подобраны приблизительно,
  стоит свериться по карте и поправить при желании.
- Пороги "горячести" (`PRICE_PER_ROOM_*`, `CENTER_*_KM`) — субъективная
  эвристика с потолка, не основана на реальной статистике рынка.
- `?debug=1` и `?resetLast=N` не защищены — любой, у кого есть URL,
  может их вызвать. Для личного некритичного проекта это осознанный
  компромисс.
- Гонка на `seen_ids` в KV между Cloudflare Cron Trigger и GitHub
  Actions при почти одновременном срабатывании (см. раздел про
  историю выше) — может изредка приводить к дублю уведомления по
  одному и тому же объявлению. Не критично, но если будет раздражать —
  нужна атомарная блокировка (например, через Durable Object), KV сама
  по себе такого не даёт.
- Эта ветка меняет `wrangler.toml`, а автодеплой Cloudflare Workers
  Builds настроен на пуш в `main` — пока ветка не смержена в `main`,
  Cron Trigger в реальном Cloudflare-проекте не появится.
