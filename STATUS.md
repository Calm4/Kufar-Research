# Kufar rental monitor — как это устроено сейчас

Сервис мониторит поиск аренды квартир на `re.kufar.by` (Гомель) и шлёт
уведомления о новых объявлениях в Telegram, с ценой, числом комнат,
расстоянием до центра и цветной оценкой "горячести".

## Архитектура

```
Внешний бесплатный пинг-сервис (cron-job.org, раз в 5 мин)
        │  обычный HTTP GET с ?token=...
        ▼
Cloudflare Worker "kufarresearch" (TypeScript, src/)  ──►  re.kufar.by (SEARCH_URL)
        │        │                                            │
        │        │                                       HTML + __NEXT_DATA__ (JSON)
        │        ▼
        │   Workers KV (KUFAR_KV): "seen_ids", "last_run", "subscribers"
        ▼
  Telegram Bot API (sendMessage, для каждого подписчика) ──► чат бота KufarChecker

Telegram (входящие апдейты) ──► POST /telegram-webhook?token=...
        │
        ▼
  /start, /price, /rooms, /filters, /clearfilters
        │
        ▼
  subscribers.ts (chatId + фильтры) → KV "subscribers"
```

Планировщик — **внешний сервис**, не Cloudflare Cron Trigger и не GitHub
Actions. Почему оба этих варианта отброшены — см. "История планировщика"
ниже. Код Worker'а переписан с одного файла (`worker.js`) на модульный
TypeScript в `src/`, с тестами на чистую логику.

## Откуда берутся данные

`re.kufar.by` рендерит страницу поиска на сервере (SSR) и встраивает все
объявления в JSON внутри `<script id="__NEXT_DATA__">`. Worker:

1. Скачивает `SEARCH_URL` (`fetcher.ts`) с заголовками обычного
   Chrome-браузера. Передаёт `cf: { cacheTtl: 0 }` — Kufar сам стоит за
   Cloudflare, без этого можно получить закэшированную версию страницы.
2. Ищет `__NEXT_DATA__`, парсит как JSON, рекурсивно ищет массив объектов
   с полем `ad_id` (`kufar.ts`, `findAdsArray`) — не завязан на точный
   путь вроде `props.pageProps...`, чтобы пережить редизайн Kufar.
3. Для каждого объявления достаёт `rooms`, `coordinates`, `address`,
   `priceUsd`/`priceByn` (см. `buildAdDetails`).
4. **Фолбэк**: если `__NEXT_DATA__` не нашёлся/не распарсился —
   `extractAds` откатывается на regex по ссылкам `/vi/{регион}/snyat/kvartiru/{id}`.
   В этом случае есть только id и ссылка, без цены/комнат/адреса.

## Логика diff и уведомлений (`monitor.ts::runMonitor`)

1. Если Kufar ответил не `200` — считается "заблокировано", в KV ничего
   не пишется (кроме `last_run` со статусом `blocked: true`), отчёт
   показывает статус и кусок ответа.
2. Иначе — собираются все id со страницы (обычно ~30, свежие сверху).
3. Читается `seen_ids` из KV: если ключа нет вообще — **первый запуск**
   (база сохраняется, уведомления не шлются, иначе улетел бы залп из
   ~30 сообщений); иначе новые = найденные минус уже виденные.
4. По каждому новому id — сообщение шлётся подписчикам из KV
   (`subscribers.ts::getSubscribers`), у которых объявление проходит их
   личный фильтр (`filters.ts::matchesFilter` — цена, количество комнат;
   объявление без цены/комнат в разметке проходит фильтр всегда), с одним
   ретраем на каждого при сбое (`telegram.ts`). Ошибки не прерывают цикл,
   копятся в отчёте. Если подписчиков нет (или ни один не подошёл по
   фильтру) — уведомления просто некому слать, это видно в отчёте и в
   `/status`.
5. Запись `seen_ids` — только если что-то изменилось (экономит дневную
   квоту записи KV). После каждого прогона (успешного или
   заблокированного) в KV пишется `last_run` — доступно через `/status`.

## Формат уведомления

```
🟢 300р(100$), 2 комн., 2.7 км до центра
3-я Авиационная ул, 9, Гомель, Гомельская область
https://re.kufar.by/vi/1075685401
```

Кружок — оценка "горячести" по цене за комнату и расстоянию до
`CITY_CENTER` (константы в `src/hotness.ts`, правь напрямую в коде).

Если у объявления цена "Договорная" — Kufar отдаёт её в JSON как
`price: 0`, а не пустым полем. `kufar.ts::buildAdDetails` трактует
неположительную цену как отсутствие цены (`priceUsd`/`priceByn` = `null`),
а `hotness.ts::formatAdMessage` в этом случае пишет явно "цена не указана"
вместо вводящего в заблуждение `0р(0$)`.

## Служебные и диагностические URL

Все требуют `?token=<ADMIN_TOKEN>` (секрет, задаётся `wrangler secret put
ADMIN_TOKEN`) — без него `401`. Это единственная защита, но она реальная:
без токена никто посторонний не сможет ни дёрнуть прогон, ни прочитать
`/status`, ни сбросить `seen_ids`.

- `/?token=...` — обычный прогон: отчёт, реально шлёт в Telegram и
  пишет в KV.
- `/status?token=...` — когда последний раз реально прошёл прогон,
  какой был статус, сколько новых/ошибок. Не трогает KV/Telegram.
  Придумано специально, чтобы не лезть в Cloudflare Observability каждый
  раз, когда хочется убедиться, что фон реально работает.
- `/?debug=1&token=...` — показывает, нашёлся ли `__NEXT_DATA__` и кусок
  JSON вокруг первого найденного id. Не трогает KV и Telegram.
- `/?resetLast=N&token=...` — удаляет последние N id из `seen_ids`
  (по умолчанию 5), чтобы на следующем прогоне они снова считались
  "новыми" — способ проверить весь пайплайн без ожидания реального
  нового объявления.
- `POST /telegram-webhook?token=...` — не для ручного открытия в браузере,
  сюда стучится сам Telegram после `setWebhook` (см. `README.md`).
  Обрабатывает команды подписки/фильтров (см. ниже) и присылает
  подтверждение на каждую. Нераспознанные тексты — подсказка со списком
  команд.

## Подписчики, /start и фильтры

До этой версии получатель уведомлений был жёстко зашит в секрет
`TELEGRAM_CHAT_ID` — ровно поэтому `/start` от кого угодно кроме владельца
этого секрета ничего не делал (кода, обрабатывающего входящие сообщения,
не существовало вообще). Теперь `TELEGRAM_CHAT_ID` убран, а получатели —
это список `subscribers` в KV (`src/subscribers.ts`), пополняемый через
`/start` (`src/telegram-webhook.ts`). Владелец бота, как и любой другой
пользователь, должен один раз написать боту `/start`, чтобы тоже получать
уведомления.

У каждого подписчика в той же записи KV хранятся необязательные фильтры,
которыми управляют либо кнопками, либо командами:

- `/price 100-300` — только объявления с ценой (в $) в этом диапазоне.
  `/price off` — снять.
- `/rooms 1,2` — только объявления с указанным числом комнат (несколько
  значений через запятую; `4` = «4 и больше», см. ниже). `/rooms off` — снять.
- `/filters` — показать текущие фильтры подписчика.
- `/clearfilters` — сбросить оба фильтра.

Проверка объявления против фильтра — чистая функция
`filters.ts::matchesFilter`, покрыта тестами (`filters.test.ts`). Если у
объявления нет цены/числа комнат в разметке Kufar — оно проходит фильтр
безусловно (см. "Что делать с неполными данными" — решение осознанное:
лучше лишнее уведомление, чем молча потерянное объявление). Количество
комнат — не точное число, а "бакет" (`filters.ts::roomBucket`): 1/2/3 точно,
а всё, что ≥4, схлопывается в один бакет `4` ("4+"), и в кнопках, и при
ручном вводе `/rooms 4` (или `/rooms 7` — тоже станет "4+").

Формат хранения в KV обратно совместим: старые записи `subscribers` в
виде голых строк `chat_id` (до фильтров) по-прежнему читаются корректно
— `subscribers.ts::normalize` трактует такую строку как подписчика без
фильтров.

### UI: кнопки вместо ручного ввода команд

Помимо текстовых команд выше, есть Telegram-native UI (`src/filter-ui.ts`):

- **Постоянная нижняя панель** (`ReplyKeyboardMarkup`, `mainMenuKeyboard()`)
  с кнопками «🔍 Фильтры» и «♻️ Сбросить фильтры» — появляется после
  `/start` и остаётся видна, пока чат открыт (это персистентный UI
  Telegram уровня чата, а не конкретного сообщения).
- Тап «🔍 Фильтры» шлёт то же самое, что команда `/filters` — бот отвечает
  панелью с **inline-кнопками** (`buildFiltersKeyboard()`): переключатели
  количества комнат (`1` / `2` / `3` / `4+`, мультивыбор) и пресеты цены
  (`до 200$` / `200–400$` / `400–600$` / `600$+` / `Любая цена`, выбор
  одного варианта), плюс кнопка «♻️ Сбросить всё».
- Тап по любой из этих кнопок приходит в вебхук как `callback_query` (не
  как `message`), обрабатывается отдельно в `telegram-webhook.ts` — меняет
  фильтр в KV и **редактирует то же самое сообщение** (`editMessageText`)
  с обновлёнными галочками, вместо того чтобы слать новое сообщение на
  каждый тап.
- Диапазон цены, не попадающий ни в один пресет (например, задан вручную
  через `/price 150-350`) — валиден и работает, просто ни одна кнопка не
  подсвечивается как выбранная (кнопки — это готовые пресеты, а не
  единственный способ задать цену).

## Планировщик — внешний пинг-сервис

Настраивается вручную (я не могу сделать это за вас — нужен ваш
аккаунт): регистрируешься на [cron-job.org](https://cron-job.org) (или
аналог — healthchecks.io, EasyCron), создаёшь задачу на URL
`https://kufarresearch.calm4.workers.dev/?token=<ADMIN_TOKEN>` с
интервалом 5 минут. Точные шаги — в `README.md`.

## Cloudflare: конфигурация (`wrangler.toml`)

- `main = "src/index.ts"` — TypeScript-сборка, Wrangler компилирует сам.
- **Без `[triggers]`** — сознательно, планировщик полностью снаружи.
- `[observability] enabled = true` — логи в Observability → Logs.
- `[[kv_namespaces]]` — биндинг `KUFAR_KV`.
- `[vars]` — `SEARCH_URL`, `MAX_SEEN_IDS`.
- Секреты (`wrangler secret put`, не в файле): `TELEGRAM_BOT_TOKEN`,
  `ADMIN_TOKEN`. (`TELEGRAM_CHAT_ID` больше не нужен — можно удалить
  через `wrangler secret delete TELEGRAM_CHAT_ID`, получатели теперь
  из `subscribers` в KV.)

Деплой автоматический: пуш в `main` пересобирает и выкатывает Worker
через Cloudflare Workers Builds. Локально — `npm run deploy`.

## История планировщика: cron → GitHub Actions → cron → внешний пинг

Три итерации, каждая обнаружила свой отдельный failure mode:

1. **Cloudflare Cron Trigger (изначально).** Запросы к Kufar,
   инициированные Cron Trigger, стабильно получали `403` ("Доступ
   ограничен") — тот же код, вызванный обычным HTTP-запросом, всегда
   получал `200`. Похоже, Cron Triggers исполняются из отдельного,
   ограниченного пула IP, который Kufar заблокировал.
2. **GitHub Actions** (`schedule: cron`). Перенесли планирование туда —
   обычный внешний `curl`, неотличимый от визита из браузера. Проблема:
   диагностика через GitHub API (`list_workflow_runs`) показала — у
   workflow за всё время существования был **ровно один** прогон, и это
   ручной `workflow_dispatch`. Ни одного автоматического `schedule`-
   события GitHub не прислал вообще, хотя прошло значительно больше
   5-минутных интервалов, чем нужно для нескольких срабатываний.
   `schedule` в GitHub Actions — best-effort, без гарантии срабатывания.
3. **Возврат Cloudflare Cron Trigger** как временная мера — сработал, но
   с тем же периодическим риском `403`, который никуда не делся: это не
   баг кода, а поведение Kufar/Cloudflare на уровне инфраструктуры.

Итог: обе "внутренние" платформы (Cloudflare Cron Trigger, GitHub Actions
schedule) оказались либо ненадёжны по срабатыванию, либо подвержены
блокировке по IP. Решение — **вынести планирование полностью наружу**:
внешний бесплатный пинг-сервис делает обычный HTTP-запрос с "гражданского"
IP-пула, не привязанного ни к одной из этих инфраструктур, и настраивается
один раз в веб-интерфейсе, а не через код/деплой.

## Известные ограничения / что можно доточить

- `CITY_CENTER` — координаты центра Гомеля подобраны приблизительно,
  стоит свериться по карте.
- Пороги "горячести" (`PRICE_PER_ROOM_*`, `CENTER_*_KM`) — субъективная
  эвристика, не основана на реальной статистике рынка.
- Если Kufar снова начнёт банить конкретный IP-пул (в этот раз —
  внешнего пинг-сервиса) — код сам ничего не чинит (обход WAF вне рамок
  этого воркера), но и не ломается: заблокированные прогоны просто не
  пишут в KV и не шлют уведомления, следующий успешный прогон подхватит
  всё, что накопилось. Проверяется через `/status`.
- Нет атомарной блокировки на запись `seen_ids` в KV — при единственном
  внешнем источнике опроса это не проблема (нет параллельных гонок),
  но если в будущем добавится второй планировщик — вернётся риск
  дублей, описанный в старых версиях этого файла.
- Нет команды отписки (`/stop`) — `telegram-webhook.ts` реагирует только
  на `/start`. Отписаться сейчас можно только вручную через Workers KV
  (или заблокировав бота в Telegram — тогда рассылка на этот чат будет
  просто попадать в `telegramErrors` и ничего не ломать).
